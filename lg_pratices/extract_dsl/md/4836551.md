## Resolution

- Please upgrade the kernel to kernel-3.10.0-693.el7 or later (the latest RHEL7.7 kernel is the most recommended) and check if the issue is resolved and never comes back after that.


## Root Cause

- The kernel crashed due to a hard lockup.
- A 'hardlockup' is defined as a bug that causes the CPU to loop in kernel mode with interrupts disabled for more than 10 seconds without letting other interrupts have a chance to run. The current stack trace is displayed upon detection and the system will crash, which is done through a sysctl 'hardlockup_panic'. The period of the hrtimer is 2*watchdog_thresh/5, which means it has two or three chances to generate an interrupt before the hardlockup detector kicks in.In this case, an exact cause of hard lockup is unclear at this time.
- We are not 100% sure of the exact root cause of the hard lockup at this time.
- In this case, it may be possible that a bug fixed upstream with a patch from commit b606f189c7d was causing the host to delay for large amounts of time in emulating LAPIC TSC deadline timer interrupt.
