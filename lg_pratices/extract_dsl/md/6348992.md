## Resolution

- Please consider migrating to RHEL8 since RHEL6 is EOL product.
- If that's not your option, please blacklist the 3rd party module and check if the issue persists after that.
- Also, please contact the hardware vendor and run the complete hardware diagnostics.
- Also we would strongly recommend upgrading the BIOS firmware to the latest.

## Root Cause

- The exact root cause is unknown at this time.
- The hard lockup occurred on the CPU on which the task is looping in the spinlock in kernel mode without giving other tasks a chance to run for more than 60 seconds.
- At that time another task had taken that spinlock and was running CPU idle routine.
- At that time, almost all CPUs' tvec_base values were way behind the current jiffies value with the only exception of the one CPU on which the hard lockup occurred.

This number is composed of two 16-bit portions: the upper value 0xfc5b and lower value 0xfc59.  
When the 2 16-bit halves have the same values, (e.g. if the spinlock value was 0xfc5bfc5b), then it means the spinlock is unlocked.  
Subtracting the lower 16bit value from the upper 16bit value tells you how many callers are interested in the spinlock.  
In this Case  2.  So there are 2 interested callers: one caller who owns it, and another two are waiting to acquire it.

The ticket-spinlock works by the spin_lock functions atomically reading the spinlock value and incrementing the upper 16bit number.  
The upper 16 bit value before the increment becomes its "ticket".  
When the lower 16bit number equals the "ticket", then that call to a spin_lock function owns the lock and returns.  spin_unlock then increments the lower 16bit number to let someone else get it.

This implementation in _spin_lock_irqsave() saves its "ticket" value in RAX, which has a value of 0xfc5a.  Since the lower 16bits of the spinlock does not equal , we know some other caller owns the lock and task is stuck waiting for someone else to release it.

This number is composed of two 16-bit portions: the upper value 0xfc5b and lower value 0xfc59.  
When the 2 16-bit halves have the same values, (e.g. if the spinlock value was 0xfc5bfc5b), then it means the spinlock is unlocked.  
Subtracting the lower 16bit value from the upper 16bit value tells you how many callers are interested in the spinlock.  
In this Case  2.  So there are 2 interested callers: one caller who owns it, and another two are waiting to acquire it.

The ticket-spinlock works by the spin_lock functions atomically reading the spinlock value and incrementing the upper 16bit number.  
The upper 16 bit value before the increment becomes its "ticket".  
When the lower 16bit number equals the "ticket", then that call to a spin_lock function owns the lock and returns.  spin_unlock then increments the lower 16bit number to let someone else get it.

This implementation in _spin_lock_irqsave() saves its "ticket" value in RAX, which has a value of 0xfc5a.  Since the lower 16bits of the spinlock does not equal , we know some other caller owns the lock and task is stuck waiting for someone else to release it.
