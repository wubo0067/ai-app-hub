## Root Cause

Two or eventually more CPUs are mutually blocking each other. Any of them can report a lockup.

When scheduling out the migration/n thread and becoming idle, the CPU tries to load balance and pull in some task from other (busy) CPU. For that it needs to grab a cpu_stopper.lock of that other CPU. If that other CPU is holding it's own lock at the same time and trying to wake up the first CPU's migration thread (the one which is just going out), it waits for the context switch to finish first before progressing further and finally releasing the lock. In other words, this a chicken/egg problem, a deadlock situation.

This issue does not exist on RHEL6, as it does not call do_numa_page() from page fault.


## Diagnostic Steps

The panic messages as in the Issue paragraph above should be sufficient. Having a vmcore memory dump one can check both CPUs' details. One CPU should be scheduling out 'migration/x' thread executing stop_one_cpu_nowait(). The other CPU participating in the deadlock should be executing stop_two_cpus() function most likely called from migrate_swap() one while handling page fault.
