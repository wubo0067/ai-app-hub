## Issue

- First indication of problem in messages: "qla2xxx 0000:0f:00.0: Mailbox command timeout occured. Scheduling ISP abort."
- After above error, the system can no longer see SAN connections or reload qla2xxx drivers
- qla2xxx recovery process fails to recover adapter at this point. Unload/reload of the driver works, but in XEN environment this fails due to memory allocation problem at load time (not enough contiguous memory available within kernel)
- When able to unload/reload driver problems is fixed

## Resolution

- This appears to be a hardware related issue of some type, contact your hardware vendor for further diagnosis.


## Root Cause

The root cause is a wedged adapter - in which the adapter stops responding to commands properly.

The first "command mailbox timeout" message indicates that 30s has elapsed since attempting a mailbox command to the adapter. Recovery is started, but it fails. Recovery contains multiple phases (loop reset, device reset, adapter reset), but during the adapter reset attempt, the reload of the firmware fails.  The firmware load and all other actions utilize the same mailbox interface (pci registers) into the adapter for communication.

It is currently unknown why an unload/reload addresses the issue. This is possibly due to IRQ platform h/w issues a re-load of driver results in re-registering IRQs -- but this is merely one of many possibilities.


## Diagnostic Steps

Offered instrumented driver that will do the following at the initial point of failure (mailbox command timeout) to gather additional information before creating BZ on issue.* output state information from hba software state data structure* output pci interface register content from hba* output mailbox command attempted* output stack back trace* expanded debug information from dpc thread (recovery thread within driver) to track its actions more closely.* Check the FC error statistics :


```
~~~
    cat /sys/class/fc_host/host*/statistics/{error_frames,invalid_crc_count,seconds_since_last_reset}
    ~~~
```

`~~~
    cat /sys/class/fc_host/host*/statistics/{error_frames,invalid_crc_count,seconds_since_last_reset}
    ~~~`The system eventually panics due to the NMI watchdog detecting a hung CPU.  This is due to interrupts and preemption being disabled on the CPU for more than 30 seconds.  The CPU is stuck executing code within this loop:


```
static inline void
qla24xx_reset_risc(scsi_qla_host_t *ha)
...
    /* Wait for soft-reset to complete. */
    d2 = RD_REG_DWORD(&reg->ctrl_status);
    for (cnt = 6000000 ; cnt && (d2 & CSRX_ISP_SOFT_RESET); cnt--) {
        udelay(5);
        d2 = RD_REG_DWORD(&reg->ctrl_status);  <-- was here at the time of the panic
        barrier();
    }
```

`static inline void
qla24xx_reset_risc(scsi_qla_host_t *ha)
...
    /* Wait for soft-reset to complete. */
    d2 = RD_REG_DWORD(&reg->ctrl_status);
    for (cnt = 6000000 ; cnt && (d2 & CSRX_ISP_SOFT_RESET); cnt--) {
        udelay(5);
        d2 = RD_REG_DWORD(&reg->ctrl_status);  <-- was here at the time of the panic
        barrier();
    }`This loop has the potential to do 6000000 iterations each with a 5 usec delay which is 30 seconds (plus processing overhead) and that's the threshold for the NMI watchdog.  So the hardware had not completed the soft reset for at least 30 seconds.  While this loop was being executed the CPU had a spinlock held with interrupts disabled.  I think the loop counter needs to be reduced to keep the processing time under the NMI watchdog threshold.  The root cause though is the non-responding hardware.
