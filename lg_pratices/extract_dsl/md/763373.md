## Resolution

Update to the kernel revision applicable, or newer, from the particular release stream below:*RHEL-6.6kernel-2.6.32-504.el6 per errata releaseRHSA-2014-1392*RHEL-6.5.zkernel-2.6.32-431.17.1.el6 per errata releaseRHSA-2014-0475*RHEL-6.4.zkernel-2.6.32-358.42.1.el6 per errata releaseRHSA-2014-0634


## Root Cause

This is due to the post_schedule_rt() function entering a memory allocation request that could block in waking the kswapd kernel thread. This caused the system to enter a deadlock state to which it cannot recover as each spinlock is being held waiting for the other to release.

Similar to the following diagram:


```
CPU31                                    CPU 1     
+-------------+   Requesting rq->lock()  +-------------+
|             |  <--------------------+  |             |
|   Holding   |                          |   Holding   |
|  rq->lock() |                          |  rq->lock() |
|             |                          |             |
|             |  +-------------------->  |             |
+-------------+   Requesting rq->lock()  +-------------+
```

CPU 1 has gone through double_lock_balance() so it's done the right thing and followed the locking hierarchy.
