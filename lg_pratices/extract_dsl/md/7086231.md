## Environment

- Red Hat Enterprise Linux 8
- Serial console


## Issue

- Require assistance into diagnosing why this node kernel panicked.
- CPU Hard Lockup occurred when system was writing to the serial console


```
"Kernel panic - not syncing: Hard LOCKUP"
```

`"Kernel panic - not syncing: Hard LOCKUP"`
## Resolution

- Reduce serial console log level/messages from current 15 to 3:

The current console log level is 15. By changing the console log level to 3, lower-level messages lower than error will no longer be written to the very slow serial console, and this will probably work as a workaround.


```

        # echo "3 4 1 7" > /proc/sys/kernel/printk
```

- It is recommended to contact hardware vendor (Dell support in this case) and request a comprehensive hardware diagnostic.Diagnostics and isolation from the perspective of a faulty CPU: As mentioned, the rcu_sched CPU stall occurred on CPU 15 in the first place, and during the dump stack processing, while writing to the very slow serial console, the rnp->lock spinlock was held. As a result, CPUs 3 and CPU 8, which were waiting for the lock, became locked up.

It's important not to overlook the fact that the rcu_sched CPU stall occurred not while CPU 15 was writing messages to the serial console, but while it was simply idling, which might suggest a faulty CPU. It is recommended to contact hardware vendor (Dell support in this case) and request a comprehensive hardware diagnostic. Please note that it is not uncommon that hardware vendors are unable to find any faulty CPUs through hardware diagnostics. We would recommend offlining the potentially faulty CPU and its sibling if that's the case.

## Root Cause

- System was writing excessively to the very slow serial console.
- Also there is a possibility of a faulty CPU.
