## Environment

- Red Hat Enterprise Linux 7.4 based hosts
- kernel version 3.10.0-693.el7 or errata kernel such as in the example under 'Diagnostic Steps'
- Windows guests with more than one virtual CPU
- the HyperV enlightenment featurehv_timeis enabled for Windows guests (default)
- Red Hat Virtualization (RHV) 4.1

`hv_time`
## Issue

- Physical host rebooted.
- Hypervisor crashes occasionally during VM migration.
- A 'Kernel panic - not syncing: Hard LOCKUP' can occur on a KVM Hypervisor after update to RHEL7.4 kernel.


## Resolution

- The issue was fixed in kernel-3.10.0-693.8.1.el7, which was shipped in RHVH 4.1.8. Please upgrade.
- This bug was tracked byBZ#1513115and was fixed in kernel-3.10.0-862.el7 viaRHSA-2018:1062.

## Root Cause

Aqemu-kvmprocess provides the execution environment for a KVM virtual machine. Each virtual CPU (vCPU) is implemented as a separate thread of the process. The panic is caused by a deadlock of two vCPU threads of the sameqemu-kvmprocess that are executing code in the KVM kernel modules that is related to the paravirtualized clock source and time reference mechanism. Since the deadlock can typically occur while one of the two vCPU threads is reading the virtualHV_X64_MSR_TIME_REF_COUNTregister, this issue is only seen in connection with Windows guests where thehv_timefeature is enabled. The command line of theqemu-kvmprocess would contain a CPU option string similar to this example.

`qemu-kvm``qemu-kvm``HV_X64_MSR_TIME_REF_COUNT``hv_time``qemu-kvm`
```
-cpu SandyBridge,hv_time,hv_relaxed,hv_vapic,hv_spinlocks=0x1fff
```
