## Environment

- Red Hat Enterprise Linux 6.10.z (kernel-2.6.32-754.18.2.el6)
- Intel(R) Xeon(R) CPU E5-2650 v4
- HPE ProLiant DL380 Gen9


## Issue

- The kernel crashes due to a hard lockup occurs on a CPU looping in a cputimer lock spinlock which is to lock for fields in thread_group_cputimer struct.

## Resolution

- A resolution is not found out at this time.
- Please elaborate on the steps to reproduce the issue if possible.
- Since RHEL6 is in the final days ofMaintenance Support 2 Phaseand moves intoEOLsoon, we would strongly recommend migrating to RHEL8 and check if the issue resolves and never comes back.
- If you are willing to troubleshoot the issue with RHEL6.10.z kernel running, pleaseinstall the latest RHEL6.10.z kerneldisable C-statesdisable a kernel sysctl knob "kernel.hardlockup_panic"enable a kernel sysctl knob "kernel.hardlockup_all_cpu_backtrace"and thenrun NMI to crash the kernelthat is getting unresponsive with massive hard lockup messages and CPU backtracesand provide a new vmcore file.

- install the latest RHEL6.10.z kernel
- disable C-states
- disable a kernel sysctl knob "kernel.hardlockup_panic"
- enable a kernel sysctl knob "kernel.hardlockup_all_cpu_backtrace"
- and thenrun NMI to crash the kernelthat is getting unresponsive with massive hard lockup messages and CPU backtraces
- and provide a new vmcore file.


## Root Cause

- A root cause is unknown at this time.
- Previously we have observed thatunconditional hard lockups are sometimes encountered on a server running with Intel Xeon E3/5 v3 processors
