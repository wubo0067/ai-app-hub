## Resolution

The fix was a fairly simple modification to use GPF_ATOMIC flag when allocating memory in_scsih_smart_predicted_fault()function from

`_scsih_smart_predicted_fault()`
```
event_reply = kzalloc(sz, GFP_KERNEL);
```

`event_reply = kzalloc(sz, GFP_KERNEL);`to


```
event_reply = kzalloc(sz, GFP_ATOMIC);
```

`event_reply = kzalloc(sz, GFP_ATOMIC);`Please update tokernel-3.10.0-1127.el7as part of theErrata RHSA-2020:1016


## Root Cause

The_scsih_smart_predicted_fault()function in mpt3sas driver was using GFP_KERNEL allocation flag

`_scsih_smart_predicted_fault()`
```
/* insert into event log */
        sz = offsetof(Mpi2EventNotificationReply_t, EventData) +
             sizeof(Mpi2EventDataSasDeviceStatusChange_t);
        event_reply = kzalloc(sz, GFP_KERNEL);
```

`/* insert into event log */
        sz = offsetof(Mpi2EventNotificationReply_t, EventData) +
             sizeof(Mpi2EventDataSasDeviceStatusChange_t);
        event_reply = kzalloc(sz, GFP_KERNEL);`The calling context using memory allocation with GPF_KERNEL flag must be allowed to sleep. However_scsih_smart_predicted_fault()function using GFP_KERNEL allocation flag is called by_scsih_io_done()from interrupt context. Such flag is inappropriate, going to sleep in interrupt context will lead to various problems soon after, most likely hangs where the CPU will be stuck in interrupt context indefinitely.

`_scsih_smart_predicted_fault()``_scsih_io_done()`In this example one task that is actually behind the root cause was handling an interrupt for IO completion with a call chain:

_base_interrupt->_scsih_io_done->_scsih_smart_predicted_fault->zalloc(GPF_KERNEL)
