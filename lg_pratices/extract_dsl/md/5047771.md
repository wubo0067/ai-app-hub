## Resolution

- RHEL7:Updatekernelto3.10.0-1160.el7(released withRHSA-2020:4060) or later, which contains fixes for this issue. The fixes are already included in RHEL7.9GA.
- RHEL7.7.z:Updatekernelto3.10.0-1062.40.1.el7(released withRHSA-2020:5206) or later, which contains fixes for this issue.
- If a similar issue is observed please raise a case with Red Hat support for further investigation.

`kernel``3.10.0-1160.el7``kernel``3.10.0-1062.40.1.el7`
## Root Cause

This is a race condition which is not likely to be hit, and it is dependent on having the system under memory pressure and having kernel code that forces a page fault within user address space while running within an atomic critical section (with interrupts disabled). Some 3rd party vendors whose kernel modules/daemons are running are capable of creating such rare deadlock described scenario.

This issue was investigated in Red Hat viaInternal Bugzilla 1799224. The issue can be fixed by backporting commit [b13b1d2d8692b437203de7a404c6b809d2cc4d99]x86/mm: In the PTE swapout page reclaim case clear the accessed bit instead of flushing the TLB.

`x86/mm: In the PTE swapout page reclaim case clear the accessed bit instead of flushing the TLB`- Kernel may face deadlock between page fault handler and __get_free_pages():


```
On one core:

page fault > page_fault -> do_page_fault -> __do_page_fault -> handle_mm_fault -> handle_pte_fault -> _raw_spin_lock
```

`On one core:

page fault > page_fault -> do_page_fault -> __do_page_fault -> handle_mm_fault -> handle_pte_fault -> _raw_spin_lock`The spinlock is locked by another core, we are spinning here with disabled interrupts.


```
On another core:

__get_free_pages -> ... -> __alloc_pages_slowpath -> try_to_free_pages -> ... -> shrink_zone -> ...  -> page_referenced -> rmap_walk -> page_referenced_one -> ptep_clear_flush_young -> flush_tlb_page -> ... -> smp_call_function_single -> generic_exec_single
```

`On another core:

__get_free_pages -> ... -> __alloc_pages_slowpath -> try_to_free_pages -> ... -> shrink_zone -> ...  -> page_referenced -> rmap_walk -> page_referenced_one -> ptep_clear_flush_young -> flush_tlb_page -> ... -> smp_call_function_single -> generic_exec_single`- The spinlock was locked bypage_check_address()inpage_referenced_one(), after this IPI was sent to the first core and this core is waiting for IPI completion ingeneric_exec_single().
- The IPI can't be delivered to the first core because it disabled interrupts while looping on spinlock.

`page_check_address()``page_referenced_one()``generic_exec_single()`The page fault handler can race against the rmap code causing a deadlock aroundthe PTL spinlock and the IPI-based TLB shootdown:
