## Resolution

- It's quite highly likely that a bug on the module named "event_collector_2_0_324607" which is not shipped with RHEL is causing an rq lock spinlock deadlock and thus a hard lockup.
- Please engage the module provider to check for some known module bug and a resolution.
- Unfortunately, sincewe Red Hat neither provide support service nor have access on the module source that is not shipped with RHEL, we do not have much to diagnose the issue caused by the module bug further.


## Root Cause

- One task calls wake_up_new_task() and takes an rq lock spinlock to wake up a task and then calls kmem_cache_alloc() to allocate an object in slab cache in its enqueue_task callback path.
- Since kmem_cache_alloc() is called without ___GFP_NO_KSWAPD, wakeup_kswapd() is called in the page allocation slowpath and then the task is getting stuck looping in the rq lock spinlock that has been taken in wake_up_new_task() by the task itself in the same context previously.
- Very highly likely, it's an rq lock deadlock caused by the module bug.
