## Resolution

- Update to kernel-2.6.32-504.el6 fromRHSA-2014-1392or later for a fix for this issue.


## Root Cause

- Watchdog panic-ed the system because all the CPUs where hung on a spinlock deadlock of the ratelimit_lock structure.The first half of the deadlock was caused by having low free memoryAs a result of the low memory we needed to allocate a new sk_buff structure from a full slab “size-2048” for network activityAt this point we got a page allocation error trying to get the memory for the slabNow we try to log the error to the system console going through the printk_ratelimit code.Due to the use of netconsole, we need to copy system console data to another system. This then needed a sk_buff structure be allocatedAnd now we encounter a Catch-22 situation that hits the same sk_buff page allocation error while also trying to log it to the system console again going through printk_ratelimit code on the same CPU.
- kmem showing the memory pressure at the time of the incident

Watchdog panic-ed the system because all the CPUs where hung on a spinlock deadlock of the ratelimit_lock structure.

- The first half of the deadlock was caused by having low free memory
- As a result of the low memory we needed to allocate a new sk_buff structure from a full slab “size-2048” for network activity
- At this point we got a page allocation error trying to get the memory for the slab
- Now we try to log the error to the system console going through the printk_ratelimit code.
- Due to the use of netconsole, we need to copy system console data to another system. This then needed a sk_buff structure be allocated
- And now we encounter a Catch-22 situation that hits the same sk_buff page allocation error while also trying to log it to the system console again going through printk_ratelimit code on the same CPU.
