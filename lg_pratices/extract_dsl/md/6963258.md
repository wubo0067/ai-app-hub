- Red Hat Enterprise Linux 9

## Issue

- Kernel crashes with "hard LOCKUP" while the panicking CPU is waiting for a runqueue spinlock or workqueue spinlock. (See few example backtraces in Diagnostic Steps.)
- The core of the issue is a CPU in deadlock scenario for it's own runqueue lock between update_blocked_averages() and try_to_wake_up():

## Root Cause

There are 2 factors causing this problem:

- A call to printk() from scheduler code while runqueue lock is being held causes a self-deadlock.
- A warning condition in update_blocked_averages() is satisfied, calling printk() to print the warning message with runqueue lock held.

The former is an underlying issue that needs robust rework even on upstream. Upstream consensus against "bandaid patches" is that such warnings shouldn't get reported in the first place (i.e. warning conditions should get fixed), hence a greater rework of printk is the chosen way.

The latter is an identified problem for which our engineering team has identified a potential fix:


```

Author: kuyo chang <kuyo.chang@mediatek.com>
Date:   Thu Apr 14 17:02:20 2022 +0800

    sched/pelt: Fix attach_entity_load_avg() corner case

    The warning in cfs_rq_is_decayed() triggered:

        SCHED_WARN_ON(cfs_rq->avg.load_avg ||
                      cfs_rq->avg.util_avg ||
                      cfs_rq->avg.runnable_avg)

    There exists a corner case in attach_entity_load_avg() which will
    cause load_sum to be zero while load_avg will not be.

    Consider se_weight is 88761 as per the sched_prio_to_weight[] table.
    Further assume the get_pelt_divider() is 47742, this gives:
    se->avg.load_avg is 1.

    However, calculating load_sum:

      se->avg.load_sum = div_u64(se->avg.load_avg * se->avg.load_sum, se_weight(se));
      se->avg.load_sum = 1*47742/88761 = 0.

    Then enqueue_load_avg() adds this to the cfs_rq totals:

      cfs_rq->avg.load_avg += se->avg.load_avg;
      cfs_rq->avg.load_sum += se_weight(se) * se->avg.load_sum;

    Resulting in load_avg being 1 with load_sum is 0, which will trigger
    the WARN.
```

## Diagnostic Steps

The kernel panics due to "Hard lockup" while the panicking CPU is waiting for a runqueue lock or a workqueue lock (since both are locked by the deadlocked process). The panicking task, which waits for the runqueue lock or workqueue lock, waits transitively for the deadlocked task with the following piece of backtrace (the following can also be the panicking task).

In order to confirm that this is the same problem as described by this KCS, you should find the deadlocked task with the following piece backtrace!
