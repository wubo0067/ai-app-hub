## Root Cause

- Thelpfcdriver's completion queuepring->txcmplqwas suffering a list corruption. Usage of this corrupt list contents eventually results in crash because theiocbfree list gets crossed linked with thepring->txcmplq.  Most systems will run for a while after the corruption, but will eventually crash when a scsi eh reset occurs and thetxcmplqis attempted to be flushed. The flush gets stuck in an endless loop.The problem is the abort handler does not hold thesli4ring lock while validating the IO, so the IO could complete while the driver is still preparing the abort. The erroneously generated abort, when it completes, has pointers to the original IO that has already completed, and the IO manipulation (for the second time) corrupts the list. Later the repeated use of this corruptpring->txcmplqlist results in hard lockup and crash. This bug is being tracked through private BZ#1554777.
