## Issue

A Red Hat Enterprise Linux 6 based KVM host/hypervisor can hang or incur a


```
Kernel panic - not syncing: Watchdog detected hard LOCKUP on cpu ...
```

`Kernel panic - not syncing: Watchdog detected hard LOCKUP on cpu ...`because a thread can erroneously attempt to acquire the run queue lock of a CPU recursively. This issue was originally observed with host kernel version 2.6.32-358.2.1.el6.x86_64 and MS Windows 7 guests running onAMD Opteron(TM) Processor 6272that supports thexsavefeature for floating point unit (FPU) state.

## Root Cause

Aqemu-kvmprocess provides the execution environment for a KVM virtual machine. Virtual CPUs (vCPUs) are implemented via separate threads within theqemu-kvmprocess (vCPU threads).

`qemu-kvm``qemu-kvm`A guest operating system can use thexsetbvinstruction to modify the content of a CPU'sxcr0(extended control register 0). Thexsetbvinstruction is emulated by the KVM kernel modules which perform the modification on behalf of the guest. Hence, the content of a CPU'sxcr0can vary while a vCPU thread is executing guest code on a host CPU. When a vCPU thread relinquishes the CPU (for example, when its time quantum expires), the KVM kernel modules must take care that the CPU'sxcr0is reloaded with the content that is required by the host kernel. However, under certain circumstances the KVM kernel modules miss to reloadxcr0and a subsequentxrstorinstruction (restore saved FPU state from memory to CPU) can fail. If this failure occurs during a context switch from a thread that schedules out of a CPU to a thread that schedules in to the CPU, the latter thread can erroneously attempt to acquire the CPU's run queue lock recursively (as shown in the stack traces under Diagnostic Steps).
