## Resolution

Check the RAM with the hardware vendor for faults, or replace the system RAM.

To prevent this in future, do not load EDAC kernel modules on Cisco UCS hardware.

Use the following steps to prevent thesb_edacmodule from loading:

`sb_edac`- How do I prevent a kernel module from loading automatically?


## Root Cause

The unloading of EDAC modules might occur when the hardware detects a RAM fault.

There is a problem unloading EDAC modules on Linux. That might be resolved by upstream commit:

- EDAC: Balance workqueue setup and teardown

Due to the rare occurrence of this issue, the late Extended Lifecycle Phase of RHEL 6/7, and the workaround of not loading EDAC modules, this was deemed unsuitable for fix during ELS.

Cisco advise EDAC should not be used on Cisco UCS hardware:

- Cisco Bug: CSCvf14938 - OS EDAC Module prevents UCS hardware detection and correction UCS-B series Redhat

When an issue with a DIMM occurs (ECC/UECC), the CPU register retains the event information until it is queried.  Once queried, the register releases that information permanently.  This is by design.  The EDAC module queries and clears the register and it prevents any DIMM related issues from ever logging within the Cisco logs.  To prevent this, the EDAC module should be blacklisted so the DIMM related information will log within the Cisco hardware logs.
