## Resolution

- Please contact HPE, run complete hardware diagnostics, and carry out the faulty hardware replacement if needed.


## Root Cause

- It appears that CPU 52 was running the idle routine and, for no reason, stopped executing context switching with its rq.lock spinlock being held.
- At that time, CPU 0 was calling run_rebalance_domains() and getting stuck waiting for the rq.lock spinlock that CPU 52 was holding. This caused the hard lockup.
- At that time, CPU 52 didn't have any reason to create such a long delay for context switching.
- It's very likely that this was a hardware issue because not only CPU 52 but also almost all other CPUs that were associated with physical CPU 1 were not processing timer interrupts properly and then showed quite large time lags that were more than 20 seconds.
