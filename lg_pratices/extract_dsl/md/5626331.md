## Resolution

- Enable zone_reclaim_mode and increase min_free_kbytes toavoid this sort of severe fragmentation and page allocation failures.
- Add a few dozens gigabytes more RAM on the server to avoid page thrashing and see if it survives with the current splunk workload which is large memory-consuming around the peak time.Committed_AS in /proc/meminfo would be a good hit for you to confirm that the server has sufficient memory to avoid page thrashing around the peak time.
- If you would like to limit memory utilization for splunk,cgroups memory limitwould help.
- Disable serial console writing with the exceptions for some debugging purposes.

## Root Cause

- The system was running out of memory where 165 GB out of 189 GB total RAM was utilized by splunk and splunkd threads.
- OOM killing was, therefore, invoked and massive page allocation failures were encountered in network packet interrupt contexts on interfaces running with tg3 driver.
- At that time the task on CPU 17 was exitting and thus trying to take a tasklist_lock.
- At that time the task on CPU 8 had taken that tasklist_lock and was performing OOM killing and thus writing OOM killing messages in a serial console which was very slow and time-consuming.
- As a result, the hard lockup occurred on CPU 17.
