## Environment

- Red Hat Enterprise Linux 7
- Red Hat Enterprise Linux 8
- Large number of attached SAN devices


## Issue

- Tests with RHEL8 have shown a tendency to get into a hard lockup in the console card when booting with a large amount of attached SAN devices.
- The activity to the console creating large amounts of console output as the LUN scans happen may cause the system to lockup on boot due to an NMI watchdog trigger.
- The result is a hung system as kdump has not yet started.
- The NMI panic will be hidden.


## Resolution

- Addingwatchdog_thresh=30to the kernel grub line will allow up to 30s instead of the default of 10s for the scanning and console output code to settle. This has shown to prevent the nmi watchdog at 10s.

`watchdog_thresh=30`NOTE!!!Only later RHEL8 from 8.1+ have the option available on the kernel line. For  RHEL7 its available from 7.7 latest zstream and later.


## Root Cause

The console code runs with parts of the code in interrupts disabled state. When the code is looping to write out data the time taken on a CPU can exceed the kernel watchdog timeout so the NMI is triggered.

## Diagnostic Steps

- When booting with large amount of attached LUNS the system hangs in early boot. This is likely triggering the watchdog and hence the NMI.
- By addingwatchdog_thresh=30it then boots, will then validate this is the issue you are getting into.
- Other ways to debug are to boot with for examplemodprobe.blacklist=lpfcand then load thelpfcdriver manually afterkdumpis loaded. The resulting console output will often trigger the NMI but this time you will have a vmcore
