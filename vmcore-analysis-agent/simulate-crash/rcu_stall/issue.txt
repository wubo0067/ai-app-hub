以下是详细的技术原理解析：

1. 核心机制冲突
rcu_read_lock(): 标记进入 RCU 读侧临界区。RCU（Read-Copy Update）机制要求所有读者必须在一定时间内离开临界区（即调用 rcu_read_unlock()），以便系统能够检测到“静止状态”（Quiescent State）。只有所有 CPU 都经过了静止状态，当前的 RCU 宽限期才能结束，旧的内存才能被释放。
mdelay(stall_duration_ms): 这是一个忙等待函数。与 msleep 不同，mdelay 不会调度出让 CPU，而是让 CPU 在原地空转（Spinning）。代码中默认设置为 70,000 毫秒（70 秒）。
2. 触发 RCU Stall (RCU 停滞)
Linux 内核有一个 RCU CPU Stall Detector（RCU CPU 停滞检测器）。它的工作原理如下：

当代码执行 rcu_read_lock() 后，该 CPU 进入 RCU 读侧临界区。
紧接着执行 mdelay(70000)，CPU 开始死循环空转，不发生上下文切换，也不退出 RCU 临界区。
内核参数 /sys/module/rcupdate/parameters/rcu_cpu_stall_timeout 定义了 RCU 宽限期的最大等待时间（通常默认为 21 秒 或 60 秒）。
由于 mdelay 持续 70 秒，超过了默认的超时阈值。
结果：RCU 检测器发现该 CPU 长期未报告静止状态，阻碍了整个系统的 RCU 宽限期推进。内核会打印 INFO: rcu_sched self-detected stall on CPU 相关的堆栈信息。
3. 为什么会导致 Crash (Panic)?
仅仅是 Stall 通常只会打印警告（Dump Stack），但在以下情况下会直接导致系统崩溃（Panic）：

内核配置：如果内核编译时或启动参数开启了 CONFIG_PANIC_ON_RCU_STALL=y 或者 panic_on_rcu_stall sysctl 被置为 1，检测到 Stall 时内核会主动触发 Panic。
Soft Lockup：由于 mdelay 占用了 CPU 且在某些配置下（如非抢占式内核）禁止了调度，看门狗（Watchdog）也会检测到 Soft Lockup（CPU 软死锁，即 CPU 长期卡在内核态不调度）。如果配置了 CONFIG_SOFTLOCKUP_PANIC=y，这也会导致 Crash。
内存耗尽 (OOM)：由于 RCU 宽限期无法结束，所有依赖 RCU 释放的内存（例如被 kfree_rcu 释放的对象）都无法真正回收。如果系统负载高，这会导致内存迅速耗尽，最终可能导致系统崩溃。
