## 🎯 最终诊断结果

**崩溃类型**: wild pointer dereference due to DMA memory corruption

**Panic 信息**: BUG: unable to handle kernel paging request at 0000800c00000060

**故障指令**: mov 0x50(%rbp),%rdx at show_interrupts+576 (ffffffff9eb798a0)

**根本原因**: A stray DMA write from an mpt3sas controller overwrote a kmalloc-128 slab page containing an mlx5 irqaction struct, corrupting the 'next' pointer. When irqbalance read /proc/interrupts, show_interrupts dereferenced the corrupted pointer leading to a page fault at a non-canonical address.

**详细分析**:

The system crashed in show_interrupts when irqbalance (PID 4396) read /proc/interrupts. The faulting instruction attempted to load from rbp+0x50, where rbp=0x0000800c00000010 was loaded from R15+0x18 (irqaction->next). R15 (0xff1148f4a2171e80) was the head of the irqaction list for IRQ 863, owned by mlx5_core (device 0000:3b:00.0). Examination of the irqaction struct revealed complete corruption: all fields, including handler, dev_id, next, and name contained garbage, while the slab allocator still marked it ALLOCATED in kmalloc-128. Adjacent objects in the same slab page (e.g., at 0xff1148f4a2171d80 and 0xff1148f4a2171f00) exhibited similar corruption patterns, notably the repeated value 0x30030109 and 0x30030109 embedded in larger words. This value corresponds to the mpt3sas log_info code that appears hundreds of times in the kernel log seconds before the crash. The physical page underlying the slab (0x162171000) is system RAM and thus DMA-accessible. The mpt3sas controllers (8 devices, e.g., 0000:2c:00.0) were actively logging IOP events (code 0x03, subcodes 0x011d/0x0109) at the time, indicating possible firmware events or error recovery. A healthy mpt3sas irqaction (IRQ 764) showed typical adapter_reply_queue layout, confirming that mpt3sas uses per-vector reply queues. The legitimate reply queue for that controller resides at physical address 0x12b703000, distinct from the corrupted page, indicating that the stray write did not target the reply queue itself but rather an unintended address—likely a stale or misprogrammed DMA descriptor in a reply free queue or similar data structure. The crash is thus attributed to a DMA memory corruption from an mpt3sas controller.

**可疑代码位置**:
- 文件：drivers/scsi/mpt3sas/mpt3sas_base.c
- 函数：_base_process_reply_queue or reply free queue management
- 行号：unknown

**关键证据**:
- Panic string: BUG: unable to handle kernel paging request at 0000800c00000060
- RBP (corrupted pointer) = 0x0000800c00000010 loaded from irqaction->next (R15+0x18)
- R15 = irqaction for IRQ 863, owner mlx5_core (device 0000:3b:00.0)

Ensure all DMA addresses are properly validated and that reply free queue entries are not posted with stale or arbitrary physical addresses.
